# 抖音喜欢列表与多源数据采集：批量扫描与未来演进计划

这份文档记录了我们在处理「抖音喜欢列表批量扫描」时遇到的问题、确定的架构方案，以及未来面对多账号、多平台（小红书等）和增量更新时的演进路线。

## 1. 核心问题与架构决策

### 1.1 痛点分析
当用户有上千甚至上万条抖音“喜欢”记录时：
- **模拟 UI 滚动（不可行）**：如果扩展在网页端单纯依靠向下滚动到底部来触发数据加载，会消耗大量内存，极易导致浏览器崩溃；且一次性请求大量数据容易触发抖音的防爬/验证码风控。
- **全量获取耗时与中断**：从头到尾一次性扫描一旦中断，需要重新从第一条开始，浪费大量时间和带宽。

### 1.2 架构决策：游标驱动的分批扫描 (Cursor-based Batching)
通过研究扩展的网络劫持逻辑 (`dy-fetch-tap.ts`) 和抖音的 `/aweme/v1/web/aweme/like/` 接口，我们采用以下策略：
1. **直接 API 调用**：扩展直接利用拦截到的认证信息，发送带 `cursor` 的主动请求，而非被动滚动。
2. **后端状态机**：后端作为进度的大脑，将每一次请求返回的 `max_cursor` 存储到任务队列中。
3. **分批接力**：后端每次只下发有限的任务（如单次扫描 15 个游标页，约 300 条），扩展完成后上报结果与新的 `cursor`。若结果指示 `has_more` 且存在新游标，后端自动入列后续的继续拉取任务 (Continuation Task)，不消耗每日扫描预算 (`daily_budget=0`)。
4. **进度恢复**：中断后，再次点击扫描即可从上次记录的 `cursor` 处继续，实现断点续传。

## 2. 当前实施细节 (Phase 1-5 总结)

- **扩展后台 (`dy-task-dispatcher.ts`)**：增强 `TaskProgress` 接口，按 `Scope` (如 `dy_like`) 保存当前游标。
- **扩展前端通信 (`dy-fetch-tap.ts` / `douyin.ts`)**：支持从后台接收 `cursor` 并注入到抖音 Fetch 请求参数中，提取响应中的 `max_cursor` 上报。
- **后端守护 (`app.py` & `dy_tasks.py`)**：在 `/api/sources/dy/enqueue` 端点接收任务触发请求；在 `dy_task_result` 处理完毕后检查 `next_cursor`，并自动触发下一批任务。
- **用户交互 (`popup.js` / `popup.html`)**：在扩展弹窗的设置页面添加了“全量扫描喜欢”按钮，直接通过 `/api/sources/dy/enqueue` 触发带游标的初始化。

## 3. 未来演进路线

### 3.1 应对“新增的抖音喜欢”（增量更新）
- **问题**：全量扫描完成后，用户每天还会在手机上产生新的“喜欢”，如何同步？
- **方案**：不需要再带 `cursor` 进行深度扫描。扩展或桌面端只需发起“增量同步”任务：
  1. `cursor` 留空，获取最新的喜欢列表。
  2. 比对本地数据库的 `bvid` 或 `item_id`，一旦遇到已经入库的视频 ID，即刻中止该批次的拉取。
  3. 这个动作可以结合当前的自动轮询任务（如现有的 60 秒 Alarm），实现无感自动同步。

### 3.2 应对“第二个抖音号”（多账号支持）
- **问题**：目前扩展全局依赖于当前登录的抖音 Cookie，如何切换或并行多个账号？
- **方案**：
  1. **状态隔离**：后端的 `DyTaskQueue` 或 `douyin_scan` 状态管理器需要引入 `account_id`（通常使用抖音的 `sec_uid` 作为唯一主键）。
  2. **扩展改造**：`dy-fetch-tap.ts` 在拦截和主动发起请求时，必须校验当前网页登录的 `sec_uid` 与任务要求的 `account_id` 是否一致。
  3. **UI 改造**：在扩展 `popup.html` 中列出检测到的多账号列表，提供下拉框让用户选择对哪个账号发起扫描。

### 3.3 应对“小红书等新平台”的批量点赞扫描
- **通用化设计**：小红书（XHS）等平台的机制与抖音相似（通常使用 `cursor` 或 `page` 分页）。
- **实施步骤**：
  1. 确认小红书的点赞/收藏 API 及其分页参数（如 `page` 或 `cursor`）。
  2. 在 `xhs_tasks.py` 和 `xhs-task-dispatcher.ts` 中复用抖音的“任务入列 -> 执行 -> 上报 next_cursor -> 自动继续入列”的模式。
  3. 将 `popup.js` 中的扫描按钮抽象为一个 `SourceScanner` UI 组件，以便快速支持新平台。

## 4. 后续工作流

目前“第一个抖音号”的基础设施、扩展 UI 及后端任务调度已经部署并开发完毕。您可以重新加载扩展并启动后端，从扩展设置页点击「全量扫描喜欢」进行断点接力扫描验证。

### 3.4 算力解耦：本地化向量嵌入与按需远程大模型推理
- **场景**：全量获取喜欢列表后，需要对大量视频和内容进行 Embedding 向量化，并在后期提供深度分析与推荐。
- **方案**：
  1. **本地轻量级 Embedding**：充分利用本地机器的资源（如配置轻量的 `embedding 8b` 模型），针对抓取下来的海量文本和视频元数据做离线、批量的嵌入操作，并将生成的向量持久化存储在本地的向量数据库中。
  2. **按需远程调用大模型**：对于复杂的聚合分析、自然语言查询和推荐生成（如使用 `qwen3:30b` 等大参数推理模型），我们推迟到您明确发起“推荐”或“分析”指令时再进行。届时，系统会基于本地向量库检索出相关的 Top-K 数据片段，将它们拼装成 Context 后，发送给性能强大的远程服务器去处理。
  3. **优势**：该方案将重 I/O 且适合批处理的“数据预处理（向量化）”与重显存、重算力的“深层逻辑理解”隔离。这样可以避免不必要地长驻或高频调用远程服务器，并减少远程算力和带宽消耗。
